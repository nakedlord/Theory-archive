# SynTax Patch v5.4 — Version Namespace Rule

```text
layer: governance  (задаёт правило нумерации для обоих слоёв)
status: active governance patch
created: 2026-07-07
module: 02_SYNTAX/PATCHES
depends_on: v5.3_reflexive_theory_status.md
checklist: 00_META/CANDIDATE_TO_CANON_CHECKLIST.md — passed
```

## Проблема до патча

Патч v5.3 разделил SynTax на два слоя: **governance** (как ведётся архив) и **runtime** (как думает агент).

Но нумерация осталась общей, и слои имеют пересекающиеся номера:

```text
governance:  v1.9  v2.6  v2.8  v3.0  v3.3  v3.5  v3.6  v3.7  v4.3  v4.6  v4.9  v5.2  v5.3
runtime:                                                                   v5.0
```

`v5.0` (runtime) и `v5.2` (governance) выглядят как последовательные версии одного объекта. **Они не связаны.**

### Наблюдаемый вред

2026-07-07: агент, читая модуль, диагностировал «форк версий: CANON говорит v4.9, PATCHES содержит v5.2, пользователь работает по v5.0 — три текущие версии, две разошедшиеся ветки». Диагноз был **неверен** — форка нет, есть два ортогональных слоя. Ошибка потребовала отдельного аудита и была снята только чтением содержания патчей.

Коллизия нумерации **произвела ложную гипотезу у читателя модуля**. Это не гипотетический риск.

## Что вводит патч

### Правило разведения пространств имён

```text
governance-линия остаётся в 5.x и продолжается: v5.4, v5.5, ...
runtime-линия при следующем изменении уходит в 6.x: v6.0, v6.1, ...
```

После этого диапазоны не пересекаются. Номер однозначно определяет слой.

### Правило объявления слоя

Каждый файл модуля обязан объявлять слой в шапке:

```text
layer: governance   — как ведётся архив
layer: runtime      — как думает агент
layer: governance + runtime — граничный файл (редко; требует обоснования)
```

Проставлено во все файлы модуля 2026-07-07.

### Правило ссылок

При ссылке на версию указывать слой, если он не очевиден из контекста:

```text
плохо:  «в v5.0 механизм M7 инвертирован»
хорошо: «в runtime v5.0 механизм M7 инвертирован»
```

## Что было до / что становится после

**До:** номер версии не определяет слой. Читатель обязан открыть файл и понять по содержанию. Коллизия воспроизводится у каждого нового агента.

**После:** номер определяет слой (5.x = governance, 6.x = runtime). Шапка подтверждает. Ссылки называют слой явно.

## Кейс

**Кейс 1 — ложная гипотеза форка (2026-07-07).** См. раздел «Наблюдаемый вред». Агент потратил ход на диагностику несуществующего форка и записал ошибочный вывод в CSP, который затем пришлось снимать отдельным событием. Различающая сила патча: читатель более не может спутать слои по номеру.

**Кейс 2 — v5.3 как граничный патч.** `v5.3_reflexive_theory_status.md` — governance-патч, изменяющий канон **runtime**-слоя. Это первый файл, живущий на границе. Правило `layer: governance + runtime` делает границу видимой, а не молчаливой.

## Ограничение применения

Патч **не переписывает прошлое**. Все существующие номера (v1.9..v5.3, runtime v5.0) остаются как есть. Правило действует **вперёд**.

Причина: номера уже разошлись по внешним носителям — пользовательским preferences, чатам, CSP-событиям, чужим сессиям. Переименование оборвало бы передачу инструкции. Дешевле поставить указатель, чем переписать все копии.

**Это применение R-Theory к самому SynTax:** номер версии — культурная инструкция, распространившаяся за пределы архива; её принудительная замена = разрыв традиции с потерей носителей.

## Риски

1. **Перескок 5.0 → 6.0 читается как major-изменение.** Runtime может уйти в 6.0 при минорной правке, и внешний читатель решит, что произошёл перелом архитектуры. Смягчение: в `RUNTIME_SPEC.md` фиксировать `semantic_version` отдельно от `namespace_version`, если понадобится.

2. **Граничные файлы размывают модель.** Если патчей вида v5.3 (governance, меняющий runtime-канон) станет много, двухслойная модель начнёт течь. Триггер пересмотра: **три и более граничных файла** → вернуться к варианту раздельных префиксов (`gov-` / `rt-`).

3. **Правило требует дисциплины при ссылках.** Никакой механизм его не проверяет. Кандидат на автоматизацию: линтер, требующий `layer:` в шапке каждого файла модуля.
