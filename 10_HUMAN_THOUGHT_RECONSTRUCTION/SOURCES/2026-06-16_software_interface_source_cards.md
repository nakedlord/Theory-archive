- HYP: Human Thought Reconstruction inference; breaks if later source work

- MET: methodological use boundary, not a historical claim.

## Purpose

MET: These source cards support the software/interface lane of Human Thought

```text
Engelbart NLS
```

## Source Card 1: Engelbart NLS / Augmentation System

Citations:

- Douglas C. Engelbart, "Augmenting Human Intellect: A Conceptual Framework",
  Stanford Research Institute, 1962.
  URL: https://www.dougengelbart.org/pubs/augment-3906.html
- Douglas C. Engelbart and William K. English, "A Research Center for
  Augmenting Human Intellect", AFIPS Fall Joint Computer Conference, 1968,
  pp. 395-410.
  URL: https://www.dougengelbart.org/pubs/augment-3954.html
  ACM: https://dl.acm.org/doi/10.1145/1476589.1476645
- Doug Engelbart Institute, "Doug Engelbart 1968 Demo".
  URL: https://www.dougengelbart.org/mousesite/1968Demo.html

Used for:

- FACT: Engelbart's research program framed computing as augmentation of human
  intellectual work, not only calculation.
- FACT: NLS combined display workstations, text editing, hierarchical view
  control, links, mouse pointing, and five-key chord keyset control.
- FACT: The 1968 demo pages document blank-document editing, hierarchical view
  control, list rearrangement, internal linking, and graphic-file links.
- HYP: For HTR, NLS is a source anchor for the shift from stored symbol to
  actively manipulated symbol structure.

Supports:

```text
editable symbol structure
-> visible view/control options
-> pointer/keyset command selection
-> immediate workspace state change
```

Does not support:

- FACT: NLS is not the same as later consumer GUI computing.
- FACT: The sources do not prove that every NLS feature directly caused later
  interfaces.
- HYP: NLS does not by itself prove a new stage of human thought; it only
  supplies a strong interface artifact for testing HTR.

HTR confidence:

```text
supported for NLS feature cluster;
strong for interface-as-operation-surface premise;
medium for the operator name extracted in the case
```

## Source Card 2: Nelson / Xanadu Hypertext

Citations:

- Theodor H. Nelson, "Complex information processing: a file structure for the
  complex, the changing and the indeterminate", ACM '65, 1965.
  DOI: https://doi.org/10.1145/800197.806036
  Bibliographic entry:
  https://elmcip.net/critical-writing/complex-information-processing-file-structure-complex-changing-and-indeterminate
- Theodor H. Nelson, "Xanalogical Structure, Needed Now More than Ever:
  Parallel Documents, Deep Links to Content, Deep Versioning, and Deep Re-Use",
  ACM Computing Surveys 31(4es), 1999.
  URL: https://cs.brown.edu/memex/ACM_HypertextTestbed/papers/60.html
- Xanadu Australia, "What is Xanadu?"
  URL: https://www.xanadu.com.au/
- Ted Nelson, archived Xanadu page, "Basic Xanadu Objectives".
  URL: https://xanadu.com.au/ted/XU/XuPageKeio.html

Used for:

- FACT: Nelson's 1965 work is a primary hypertext source for complex,
  changing, non-linear file structures.
- FACT: Later Xanadu descriptions emphasize deep links, versioning, side-by-side
  comparison, visible reuse, transclusion, and stable addressing.
- FACT: Xanadu is a design lineage and long-running project, not a mass-adopted
  everyday interface in the way the Web became.
- HYP: Xanadu shows a maximalist form of editable symbol structure: documents
  are not isolated pages but addressable, reusable, versioned connection fields.

Supports:

```text
document
-> addressable spans / versions / links / transclusions
-> visible relation structure
-> editing and reading as navigation through durable connections
```

Does not support:

- FACT: It does not support treating Xanadu as the actually deployed Web.
- FACT: It does not support claiming that transclusion became the default
  public hypertext model.
- HYP: It does not prove that all useful interface cognition requires
  bidirectional links or transclusion.

HTR confidence:

```text
supported for hypertext design concepts;
medium for historical deployment impact;
strong as a contrast source against simpler Web pages
```

## Source Card 3: HyperCard / Cards, Stacks, Scripts

Citations:

- Apple Computer, "HyperCard User's Guide", 1987.
  PDF: https://vintageapple.org/macprogramming/pdf/HyperCard_Users_Guide_1987.pdf
- Library of Congress, "HyperCard Stack File Format".
  URL: https://www.loc.gov/preservation/digital/formats/fdd/fdd000537.shtml
- Computer History Museum, "Bill Atkinson".
  URL: https://computerhistory.org/profile/bill-atkinson/

Used for:

- FACT: HyperCard organized user-created materials into stacks and cards.
- FACT: HyperCard stacks could include buttons, fields, cards, backgrounds, and
  scripts; HyperTalk made stacks interactive.
- FACT: Bill Atkinson conceived, designed, and implemented HyperCard at Apple.
- HYP: HyperCard is a source anchor for ordinary-user editable scaffolding:
  users could assemble small symbolic/action environments from cards, buttons,
  fields, and scripts.

Supports:

```text
card
-> button / field / background / script
-> user-built navigation or action
-> local executable mini-workspace
```

Does not support:

- FACT: HyperCard was not a globally networked hypertext system by default.
- FACT: It does not support treating all HyperCard stacks as cognitive tools;
  many were ordinary applications, games, or databases.
- HYP: It does not prove that end-user programming is necessary for every later
  interface lane.

HTR confidence:

```text
supported for card/stack/script architecture;
strong for editable scaffold premise;
medium for comparison with Web and AI chat workspaces
```

## Source Card 4: World Wide Web / Web Page

Citations:

- Tim Berners-Lee, "Information Management: A Proposal", CERN, March 1989,
  revised May 1990.
  URL: https://www.w3.org/History/1989/proposal-msw.html
- Tim Berners-Lee and Robert Cailliau, "WorldWideWeb: Proposal for a HyperText
  Project", CERN, November 1990.
  URL: https://www.w3.org/Proposal.html
- CERN, "A short history of the Web".
  URL: https://home.cern/science/computing/the-birth-of-the-web/short-history-web/
- CERN, "http://info.cern.ch - home of the first website".
  URL: https://info.cern.ch/

Used for:

- FACT: The Web proposal framed hypertext as linked nodes browsable through a
  common interface across heterogeneous CERN information systems.
- FACT: The Web simplified global hypertext around browsers, servers, addresses,
  documents, and links.
- FACT: CERN identifies `info.cern.ch` as the first website.
- HYP: The Web page is a source anchor for a broad, low-friction public
  hypertext surface: less structurally rich than Xanadu, but much more widely
  deployable.

Supports:

```text
page / resource
-> link
-> browser navigation
-> shared public information surface
```

Does not support:

- FACT: The early Web does not implement Xanadu-style stable bidirectional links,
  transclusion, or universal version management.
- FACT: A web page is not automatically an editable workspace for every reader.
- HYP: It does not prove that simplicity is always cognitively superior to richer
  hypertext structure.

HTR confidence:

```text
supported for browser/link/common-interface premise;
strong for deployed hypertext carrier comparison;
medium for editable-symbol comparison because early Web editing was uneven
```

- Anthropic Engineering, "Effective context engineering for AI agents", 2025.
  URL: https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents
- Context State Protocol local repository artifacts:

  and `C:\Users\imnak\agent-playbook\04-multi-agent\session-continuity.md`.

Used for:

- FACT: ChatGPT Projects are documented as workspaces that group chats,
  reference files, and custom instructions for ongoing work.
- FACT: Prompt engineering treats instructions and examples as a way to steer
  model behavior.
- FACT: OpenAI tool docs describe model access to built-in tools, function
  calling, tool search, and remote MCP servers.
- FACT: Anthropic's context engineering article frames agent work as management
  of instructions, tools, external data, and message history across turns.
- FACT: CSP is a local/user protocol for durable cross-session context, events,
  session cards, routing, and handoffs; it is not an external product standard.
- HYP: AI chat workspaces turn language from record/navigation into a live
  operation surface where prompts, files, tools, and durable context jointly
  shape next actions.

Supports:

```text
conversation
-> instructions / files / memory / project state
-> prompt or tool selection
-> generated response or external action
-> updated workspace context
```

Does not support:

- FACT: Current AI workspace product features are unstable and vendor-specific.
- FACT: Prompt engineering is not evidence that models understand intentions in
  the same way humans do.
- FACT: CSP is local operational evidence, not independent empirical proof.
- HYP: AI chat workspaces do not prove a final interface endpoint; they are one
  current carrier in a changing software lineage.

HTR confidence:

```text
supported for current workspace/tool/context features as of 2026-06-16;
medium for durable historical comparison because the product category is young;
high as a live case for context-state and action-menu analysis
```

## Combined Source Boundary

Allowed:

- FACT: Treat NLS as an early integrated live symbol-manipulation system.
- FACT: Treat Xanadu as a rich hypertext/versioning/transclusion design source.
- FACT: Treat HyperCard as a card/stack/script scaffold for end-user executable
  symbol structures.
- FACT: Treat the Web as a simple, widely deployed public hypertext surface.
- FACT: Treat modern AI chat workspaces as current systems combining
  conversation, instructions, files/context, and tool access.
- HYP: Compare these systems as increasingly explicit operation surfaces.

Not allowed:

- MET: Do not turn the lane into a linear progress story.
- MET: Do not claim direct mind-reading from software artifacts.
- MET: Do not claim AI chat workspaces are the culmination of interface history.
- MET: Do not promote `executable action menu / live operation grammar` to canon
  without cross-lane review.
- MET: Do not treat vendor docs as stable long-term product guarantees.
